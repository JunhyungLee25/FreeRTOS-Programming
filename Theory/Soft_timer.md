## 1. 소프트웨어 타이머의 특징

- **중앙 집중 관리**: 하나의 'Timer Service Task'가 모든 타이머를 관리함.
- **적은 자원 소모**: 하드웨어 타이머 리소스를 점유하지 않고 RAM 메모리만 소모하여 효율적임.
- **높은 확장성**: MCU 하드웨어 제약 없이 수많은 타이머 인스턴스 생성이 가능함.
- **비차단(Non-blocking)**: 특정 작업의 지연 실행을 위해 태스크를 멈출 필요가 없음.

## 2. 하드웨어 타이머 vs 소프트웨어 타이머

| 구분        | 하드웨어 타이머 (HW Timer) | 소프트웨어 타이머 (SW Timer)       |
| :-------- | :------------------ | :------------------------- |
| **정밀도**   | 매우 높음 (나노/마이크로 초)   | 상대적으로 낮음 (Tick 단위 지터 발생)   |
| **개수 제한** | 하드웨어 설계에 따라 고정됨     | 힙(Heap) 메모리가 허용하는 만큼 생성 가능 |
| **실행 환경** | 인터럽트(ISR) 컨텍스트      | 타이머 서비스 태스크 컨텍스트           |
| **복잡도**   | 레지스터 설정 필요 (복잡함)    | API 호출로 생성 및 관리 (간편함)      |

## 3. 타이머 종류

1. **원샷(One-shot)** 타이머
   - **Timeout 처리**: 데이터 응답 대기 시간 제한.
   - **Debouncing**: 버튼 노이즈 제거를 위한 지연 확인.
2. **자동 재실행(Auto-reload) 타이머**
   - **Heartbeat**: 시스템 생존 신호 주기적 전송.
   - **Sensor Polling**: 일정 간격으로 센서 데이터 수집.

## 4. 소프트웨어 타이머의 주요 활용 방식

소프트웨어 타이머는 단순히 시간을 재는 것 이상의 역할을 할 수 있다.

1. **타임아웃(Timeout) 처리:**
    
    - 통신(UART, I2C 등)을 할 때 상대방이 답장이 없을 경우 무한정 기다리기보다는 이때 100ms 원샷 타이머를 걸어두고, 답장이 오면 타이머를 끄고, 안 오면 콜백에서 에러 처리를 한다.
        
2. **디바운싱(Debouncing):**
    
    - 버튼을 누를 때 발생하는 전기적 노이즈를 제거할 때 사용한다. 버튼 인터럽트가 발생하면 50ms 원샷 타이머를 시작하고, 타이머가 만료되었을 때 여전히 버튼이 눌려 있는지 확인하여 '진짜 클릭'인지 판별한다.
        
3. **상태 모니터링 및 하트비트(Heartbeat):**
    
    - 시스템이 살아있는지 확인하기 위해 1초마다 LED를 깜빡이거나 모니터링 태스크에 신호를 보낼 때 'Auto-Reload' 타이머를 사용한다.
        
4. **지연 실행(Delayed Execution):**
    
    - 어떤 동작을 한 뒤, 특정 시간이 지난 후에 후속 조치를 해야 할 때 태스크를 블록시키지 않고 타이머를 걸어둔 채 다른 일을 할 수 있다.

## 4. 핵심 주의사항

- 타이머 콜백 내에서 **Blocking API**(`vTaskDelay`, `xQueueReceive` 등) 사용 금지.
- 콜백 함수는 최대한 짧고 간결하게 작성할 것 (지연 발생 시 다른 타이머에 영향).
- `printf` 등 무거운 함수 사용 시 타이머 태스크 스택 크기(`configTIMER_TASK_STACK_DEPTH`) 확인 필수.
