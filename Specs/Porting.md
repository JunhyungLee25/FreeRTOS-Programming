## 1. 개요 
- **정의**: 특정 OS(Kernel)를 새로운 프로세서 아키텍처나 보드 환경에서 동작하도록 수정하는 과정.

## 2. **목적**
- **하드웨어 종속성 해결:** 
  CPU들은 각각의 레지스터 구조, 스택 관리 방식, 인터럽트 처리 메커니즘이 모두 다르다. OS 커널이 이러한 하드웨어의 미세한 차이에 상관없이 공통된 인터페이스로 하드웨어를 지배할 수 있도록 번역층을 만드는 것이다.
- **소프트웨어 재사용성 및 생산성 극대화:** 
  새로운 보드가 나올 때마다 OS 전체를 새로 짜는 것은 불가능하다. 따라서 OS의 심장인 ㅅ커널 소스는 그대로 재사용하고, 하드웨어와 직접 맞닿는 아주 적은 양의 코드만 수정하여 개발 시간과 비용을 획기적으로 줄인다.
- **하드웨어 추상화 제공:**
  개발자가 애플리케이션을 짤 때, 하드웨어가 STM32인지 NXP인지 고민하게 하면 안된다. 따라서 포팅을 통해 하드웨어를 추상화하여, 상위 레이어의 개발자가 어떤 하드웨어 환경에서도 동일한 API를 사용하여 안정적으로 코딩할 수 있는 환경을 구축하는 것이다.
  

## 3. 포팅 시 주요 수정 요소 
### A. 데이터 타입 정의 (`portmacro.h`)
- 해당 CPU의 레지스터 크기에 맞게 `BaseType_t`, `TickType_t` 등을 정의 (예: 32비트 CPU는 `long`).
- **Critical Section** 보호를 위한 인터럽트 제어 매크로 구현.

### B. 태스크 스택 초기화 (`port.c`)
- 새로운 태스크가 생성될 때, CPU의 레지스터들(R0~R15 등)이 스택에 어떤 순서로 저장될지 설계함.
- 이 구조가 틀리면 태스크 전환(Context Switching) 시 시스템이 크래시됨.

### C. 틱 타이머 설정 
- **SysTick 설정**: OS의 시간 기준인 `configTICK_RATE_HZ`를 맞추기 위해 하드웨어 타이머 설정.
- **인터럽트 서비스 루틴(ISR)**: 타이머 인터럽트가 발생할 때마다 `xTaskIncrementTick()`을 호출하여 OS 시간을 업데이트함.

## 4. 일반적인 포팅 작업의 진행 순서

1. 컴파일러 변경 (단, 필요한 경우에 한함)
2. 부트 코드만 하드웨어에 맞게 변경 (우선 부팅을 시켜야한다.)
3. 컴파일 및 실행
4. 코드 실행 후 문제가 발생한 곳을 확인 후 수정
5. (3),(4)를 반복하여 수행
6. 포팅 완료

## 5. 포팅 확인 체크리스트 
1. **컴파일러 설정**: 사용하는 툴체인(GCC 등)에 맞는 어셈블리 코드가 포함되었는가?
2. **메모리 배치**: 힙(Heap) 영역과 스택(Stack) 영역이 겹치지 않게 링커 스크립트가 짜여졌는가?
3. **인터럽트 우선순위**: 커널이 사용하는 인터럽트 우선순위가 하드웨어 설정과 충돌하지 않는가?
